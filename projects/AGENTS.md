# 项目概述
本目录下的所有项目都是为 AI Agent 开发的"配件"——即 **skill**,**MCP**,**TOOLS**
  - **不是**传统软件
  - **不是**给人类直接使用的应用
项目的消费者是另一个 AI Agent，运行在一个**全新的、零历史上下文的 session** 中。

# AI组件项目设计哲学

**始终从两个视角出发：**

- **使用者视角**：用户用自然语言表达意图，不应感知底层是哪些 skill 在工作、调用了几次
- **Agent 视角**：AI AGENT读取用户意图后，自主判断调用哪些 skill、以何种顺序组合，无需用户指定流程

这意味着：**skill 是原子能力，流程由 agent 编排，而非写死在 skill 里。**

## Skill 设计标准

一个好的 skill 必须满足：

1. **description 即路由**：description 要精确描述"何时用我"，agent 仅凭 description 就能正确判断是否调用
	写法标准:
	  - **何时用**：触发场景是什么(可以举例)
          - **做什么**：核心能力概括
	  - **不做什么**：明确排除的场景（防止误路由）
	  - **依赖什么**：可能依赖哪些其他功能
2. **单一职责**：一个 skill 只做自己任务内的事，不包含对上层的感知
3. **输出对 agent 友好**：输出内容要让下游 skill 或 agent 易于消费，避免只对人类可读而 agent 难以解析
4. **声明依赖，不硬编码调用**：skill 通过 description 声明依赖哪些其他组件或部分，由 agent 决定调用链，而不是在 skill 内部写死调用逻辑
5. **渐进式披露**: skill 只暴露当前任务所需的最小必要信息，优先返回核心结论、关键字段或必要片段；更详细的内容、完整引用和扩展上下文应按需提供，而不是一次性全部展开。 

# 完工前必须通过的自检清单
[ ] 1. 空上下文测试
假设一个全新 agent session，只看到这个 skill 文件，
它能正确、完整地执行吗？有没有依赖"心照不宣"的上下文？
[ ] 2. Description 路由测试
仅凭 description，agent 能准确判断：
- 什么情况下该调用我？
- 什么情况下不该调用我？
[ ] 3. 依赖声明测试
如果依赖其他 skill，是否只在 description 中声明，而没有在 skill 内部硬编码调用逻辑？
